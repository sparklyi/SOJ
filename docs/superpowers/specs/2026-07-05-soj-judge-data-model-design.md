# SOJ Judge Data Model Design

## Decision

SOJ Judge 的底层表结构需要先按长期能力一次性定好骨架，再分阶段实现。不要把 `submission_results.details jsonb` 继续扩成万能字段，也不要把 case result、manifest、可见性策略、rejudge、cache、runner 日志一点点塞回 `submissions`。

核心原则：

- `submissions` 是用户提交事实表，只保存列表页和状态机需要的摘要字段。
- `judge_attempts` 是一次判题尝试事实表，承载协议版本、runner、manifest、错误分类和可复现信息。
- `judge_case_results` 是每个 case/group 的结构化结果表，支撑 explainability、统计、rejudge 和 admin 排障。
- `submission_results` 是当前公开摘要或最终摘要，不承担完整内部证据职责。
- `visibility policy` 在服务层裁剪返回；数据库保存完整内部事实。
- ACM/OI/自测共享 judge core 结果模型，上游按场景决定展示和计分。

## Current Schema Assessment

当前 schema 已经能支撑基础 v2，但不适合作为长期 Insight Judge 底座：

- `submissions` 同时承担状态、摘要结果、错误信息和列表查询，适合保留，但不应继续扩展 case/manifest 字段。
- `submission_results.details jsonb` 过于宽泛，短期方便，长期会导致查询、索引、权限裁剪和版本迁移困难。
- `judge_tasks` 和 `submissions` 是一对一，适合当前单次判题；长期 rejudge、retry、attempt diff 需要独立 attempt 层。
- `runs` 与 `submissions` 分离是合理的，但自测也需要复用 attempt/result 模型，否则未来两条结果链路会分叉。
- `artifacts` 可继续作为对象存储索引，但需要支持 judge attempt、case output、manifest、runner log 等 owner/kind。
- `contest_problem_results` 作为 ACM scoreboard 聚合表是合理的，不能依赖 case-level 表直接算榜。

## Target Model

### Submission Layer

`submissions` 保留为用户提交主表：

- identity: `id`, `user_id`, `problem_id`, `contest_id`, `language_id`, `testcase_set_id`
- state: `status`, `score`, `time_ms`, `memory_kb`, `error_message`
- source: `source_artifact_id`
- timing: `submitted_at`, `judged_at`, `updated_at`

不新增：

- case details
- manifest details
- runner raw logs
- hidden testcase metadata
- replay/debug-only fields

必要新增可考虑：

- `current_attempt_id bigint`，指向当前生效的 judge attempt，方便 rejudge 后切换当前结果。
- `judge_mode text`，如果后续需要区分 `formal`, `self_run`, `rejudge`，但第一阶段可先放在 attempt。

### Attempt Layer

新增 `judge_attempts`，作为每一次实际判题的核心事实表。

建议字段：

```sql
CREATE TABLE judge_attempts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    submission_id bigint REFERENCES submissions(id) ON DELETE CASCADE,
    run_id bigint REFERENCES runs(id) ON DELETE CASCADE,
    task_id bigint REFERENCES judge_tasks(id),
    attempt_no integer NOT NULL CHECK (attempt_no > 0),
    protocol_version text NOT NULL,
    judge_core_version text NOT NULL,
    judge_engine text NOT NULL,
    judge_agent_id text,
    language_id bigint NOT NULL REFERENCES languages(id),
    language_runtime text,
    sandbox_backend text,
    sandbox_profile text,
    testcase_set_id bigint REFERENCES testcase_sets(id),
    testcase_set_hash text,
    checker_hash text,
    validator_hash text,
    status text NOT NULL CHECK (
        status IN ('created', 'compiling', 'running', 'checking', 'accepted', 'wrong_answer', 'compile_error', 'runtime_error', 'time_limit', 'memory_limit', 'output_limit', 'system_error', 'canceled')
    ),
    verdict text CHECK (
        verdict IS NULL OR verdict IN ('accepted', 'wrong_answer', 'compile_error', 'runtime_error', 'time_limit', 'memory_limit', 'output_limit', 'system_error', 'canceled')
    ),
    score integer NOT NULL DEFAULT 0 CHECK (score >= 0),
    time_ms integer CHECK (time_ms IS NULL OR time_ms >= 0),
    memory_kb integer CHECK (memory_kb IS NULL OR memory_kb >= 0),
    first_failed_case_index integer,
    first_failed_group text,
    compile_output_summary text,
    stderr_summary text,
    checker_message text,
    error_class text,
    error_message text,
    manifest jsonb NOT NULL DEFAULT '{}'::jsonb,
    metrics jsonb NOT NULL DEFAULT '{}'::jsonb,
    trace_id text,
    started_at timestamptz,
    finished_at timestamptz,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CHECK ((submission_id IS NOT NULL)::int + (run_id IS NOT NULL)::int = 1)
);
```

关键约束：

- `submission_id` 和 `run_id` 二选一，formal submission 和 self-run 复用同一结果模型。
- `attempt_no` 对同一个 owner 单调递增。
- `status` 是 attempt lifecycle，`verdict` 是最终判题语义；运行中 verdict 可以为空。
- `manifest` 允许保留完整复现数据，但高频查询字段必须结构化列化。

建议索引：

```sql
CREATE UNIQUE INDEX judge_attempts_submission_attempt_uidx
    ON judge_attempts (submission_id, attempt_no)
    WHERE submission_id IS NOT NULL;

CREATE UNIQUE INDEX judge_attempts_run_attempt_uidx
    ON judge_attempts (run_id, attempt_no)
    WHERE run_id IS NOT NULL;

CREATE INDEX judge_attempts_submission_finished_idx
    ON judge_attempts (submission_id, finished_at DESC)
    WHERE submission_id IS NOT NULL;

CREATE INDEX judge_attempts_run_finished_idx
    ON judge_attempts (run_id, finished_at DESC)
    WHERE run_id IS NOT NULL;

CREATE INDEX judge_attempts_status_updated_idx
    ON judge_attempts (status, updated_at);

CREATE INDEX judge_attempts_language_finished_idx
    ON judge_attempts (language_id, finished_at DESC);

CREATE INDEX judge_attempts_trace_id_idx
    ON judge_attempts (trace_id)
    WHERE trace_id IS NOT NULL;
```

### Case Result Layer

新增 `judge_case_results`，保存每个 case 或 group 的结构化结果。

建议字段：

```sql
CREATE TABLE judge_case_results (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    attempt_id bigint NOT NULL REFERENCES judge_attempts(id) ON DELETE CASCADE,
    case_index integer NOT NULL CHECK (case_index > 0),
    group_name text,
    testcase_key text,
    status text NOT NULL CHECK (
        status IN ('accepted', 'wrong_answer', 'runtime_error', 'time_limit', 'memory_limit', 'output_limit', 'skipped', 'system_error')
    ),
    score integer NOT NULL DEFAULT 0 CHECK (score >= 0),
    time_ms integer CHECK (time_ms IS NULL OR time_ms >= 0),
    memory_kb integer CHECK (memory_kb IS NULL OR memory_kb >= 0),
    exit_code integer,
    signal text,
    checker_message text,
    output_diff_summary text,
    stdout_artifact_id bigint REFERENCES artifacts(id),
    stderr_artifact_id bigint REFERENCES artifacts(id),
    created_at timestamptz NOT NULL DEFAULT now()
);
```

建议索引：

```sql
CREATE UNIQUE INDEX judge_case_results_attempt_case_uidx
    ON judge_case_results (attempt_id, case_index);

CREATE INDEX judge_case_results_attempt_status_idx
    ON judge_case_results (attempt_id, status);
```

设计取舍：

- 不存 hidden testcase 原文，只存 key/hash/摘要和 artifact 指针。
- `output_diff_summary` 是安全摘要，完整输出走 artifacts 并受权限控制。
- `skipped` 用于 early stop 后未运行 case，避免 case 统计歧义。

### Manifest And Artifact Layer

manifest 第一阶段可以保留在 `judge_attempts.manifest jsonb`，但必须满足两个条件：

- 高频过滤字段列化：`judge_core_version`, `sandbox_backend`, `sandbox_profile`, `language_runtime`, `testcase_set_hash`, `checker_hash`, `trace_id`。
- manifest JSON 仅作为复现详情，不作为主查询路径。

`artifacts.owner_type` 需要扩展：

```sql
owner_type IN ('submission', 'run', 'problem', 'testcase', 'judge_attempt', 'judge_case')
```

`artifacts.kind` 需要扩展：

```sql
kind IN (
  'source',
  'stdout',
  'stderr',
  'compile_output',
  'judge_log',
  'attachment',
  'testcase_archive',
  'manifest',
  'case_stdout',
  'case_stderr',
  'output_diff'
)
```

长期如果 artifacts 增长很大，可以按 owner/kind/time 做分区或生命周期清理，但 MVP 先保持一张表。

### Current Result Projection

`submission_results` 建议调整定位：它不是完整结果表，而是当前生效 attempt 的 summary/projection。

建议结构：

```sql
CREATE TABLE submission_results (
    submission_id bigint PRIMARY KEY REFERENCES submissions(id) ON DELETE CASCADE,
    attempt_id bigint NOT NULL REFERENCES judge_attempts(id),
    status text NOT NULL,
    score integer NOT NULL DEFAULT 0 CHECK (score >= 0),
    time_ms integer CHECK (time_ms IS NULL OR time_ms >= 0),
    memory_kb integer CHECK (memory_kb IS NULL OR memory_kb >= 0),
    first_failed_case_index integer,
    first_failed_group text,
    error_class text,
    safe_summary jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now()
);
```

如果当前 migration 已上线前可破坏式调整，建议直接改成上面形态，不保留 `id` 和泛化 `details`。如果担心改动太大，也至少把 `attempt_id`、`first_failed_case_index`、`error_class`、`safe_summary` 加上，并把 `details` 标记为 deprecated。

### Contest Projection

`contest_problem_results` 继续作为 ACM 榜单聚合表，不保存 case detail。

建议新增：

- `best_submission_id`
- `best_attempt_id`
- `last_attempt_id`

原因：

- scoreboard 只需要聚合事实，不应该 join case results。
- rejudge 后可以明确知道榜单基于哪个 attempt。
- frozen/final snapshot 可以稳定复现当时榜单依据。

`contest_score_snapshots.payload` 保留 jsonb 合理，但 payload 应记录：

- snapshot schema version
- source attempt ids 或 generation watermark
- generated reason: `manual`, `scheduled`, `contest_end`, `rejudge`

### Problem Data Check

新增 `problem_check_runs` 和 `problem_check_findings`，不要塞进 `testcase_sets`。

建议字段：

```sql
CREATE TABLE problem_check_runs (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    problem_id bigint NOT NULL REFERENCES problems(id) ON DELETE CASCADE,
    testcase_set_id bigint REFERENCES testcase_sets(id),
    status text NOT NULL CHECK (status IN ('running', 'passed', 'warning', 'failed', 'system_error')),
    score integer CHECK (score IS NULL OR score >= 0),
    summary jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_by bigint REFERENCES users(id),
    started_at timestamptz NOT NULL DEFAULT now(),
    finished_at timestamptz
);

CREATE TABLE problem_check_findings (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    run_id bigint NOT NULL REFERENCES problem_check_runs(id) ON DELETE CASCADE,
    severity text NOT NULL CHECK (severity IN ('info', 'warning', 'error')),
    code text NOT NULL,
    message text NOT NULL,
    case_index integer,
    metadata jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

这能支撑作者体验，也能避免 testcase metadata 后续被 validation 字段污染。

### Rejudge Model

新增 `rejudge_batches`，不要只靠重复写 `judge_tasks`。

建议字段：

```sql
CREATE TABLE rejudge_batches (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    scope_type text NOT NULL CHECK (scope_type IN ('submission', 'problem', 'contest', 'language', 'testcase_set', 'judge_core_version')),
    scope_id text NOT NULL,
    reason text NOT NULL,
    requested_by bigint REFERENCES users(id),
    status text NOT NULL CHECK (status IN ('pending', 'running', 'completed', 'failed', 'canceled')),
    total_count integer NOT NULL DEFAULT 0 CHECK (total_count >= 0),
    completed_count integer NOT NULL DEFAULT 0 CHECK (completed_count >= 0),
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now()
);
```

`judge_attempts` 后续可增加 `rejudge_batch_id`，让 rejudge diff 和审计有根。

## Anti-Deformation Rules

后续实现必须遵守这些规则，避免表结构变形：

1. 不把新的 judge 证据塞进 `submissions.error_message` 或 `submission_results.details`。
2. 新增高频查询能力时优先列化字段，再用 jsonb 存低频详情。
3. case-level、manifest、problem check、rejudge batch 分表，不共用一张万能 events 表。
4. `contest_problem_results` 只做 scoreboard projection，不承担判题详情存储。
5. `judge_tasks` 只做调度，不承担 attempt/result 事实。
6. Core 结果完整落库，API 展示由 visibility policy 裁剪。
7. 所有 attempt 都能关联 protocol version、judge core version、sandbox profile、language runtime、testcase set hash。
8. 隐藏 testcase 原文不进关系表，不进普通 API response。

## Implementation Order

推荐在 Phase 2 前做一次 schema 重构，顺序如下：

1. 增加 `judge_attempts`、`judge_case_results`，扩展 artifacts owner/kind。
2. 调整 `submission_results` 为 current projection，并关联 `attempt_id`。
3. 增加 service/repository 写入 attempt + cases + current projection 的事务。
4. 增加 visibility policy，从 attempt/case 表投影 safe response。
5. 增加 `problem_check_runs` / `problem_check_findings`。
6. 增加 `rejudge_batches` 和 attempt 关联。
7. 再进入 runner/sandbox/cache 相关表或字段。

## Recommended Immediate Changes

因为项目还未上线，建议允许破坏式 migration 调整：

- 直接修改 `000001_init.up.sql`，不要新增兼容迁移。
- 同步更新 sqlc queries 和 generated code。
- 增加 schema-focused tests，验证 attempt/current projection/case result 的事务一致性。
- 更新 OpenAPI 的 submission result schema，明确 public summary 和 admin diagnostics 分离。

这个设计的目标是让后续 6 个阶段都能落在稳定骨架上，而不是每做一个功能就在现有表上补字段。
